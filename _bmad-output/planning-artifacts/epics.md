# 召唤之王 - Epic Breakdown

## Overview

This document provides the complete epic and story breakdown for 召唤之王, decomposing the requirements from the PRD, UX Design if it exists, and Architecture requirements into implementable stories.

## Requirements Inventory

### Functional Requirements

FR1: 玩家可以创建或进入自己的游戏账号会话并进入《召唤之王》一期可玩版。  
FR2: 玩家可以完成首次角色初始化并获得唯一的初始角色身份。  
FR3: 系统可以为新角色一次性发放初始资源、初始幻兽与默认队伍。  
FR4: 系统可以防止同一账号重复获得首次初始化收益。  
FR5: 玩家可以重新进入已有角色并延续上次成长状态。  
FR6: 玩家可以在首次进入后直接进入一期核心玩法，而不需要先理解延期系统。  
FR7: 玩家可以从主界面进入一期核心入口，包括背包、幻兽、战斗与基础成长相关页面。  
FR8: 玩家可以在主界面识别当前最优先的成长动作或推进目标。  
FR9: 玩家可以完成首次引导并理解基础玩法闭环。  
FR10: 玩家在成长受阻时可以获得明确的阻断提示与恢复路径。  
FR11: 系统可以向玩家提示可领取奖励、可执行养成动作或可继续挑战的内容。  
FR12: 系统可以在一期阶段隐藏、关闭或标记延期系统入口，避免干扰主循环。  
FR13: 玩家可以查看自己当前拥有的基础资源、消耗品与一期相关物品。  
FR14: 玩家可以区分不同类型的存储空间，包括背包与幻兽相关存储。  
FR15: 系统可以在玩家获得奖励时校验对应存储空间是否可用。  
FR16: 玩家在容量不足时可以收到明确提示，并知道如何处理后再继续领取或推进。  
FR17: 玩家可以通过一期允许的方式消耗资源以完成基础成长动作。  
FR18: 系统可以记录玩家核心资源的获得、消耗与结算结果，供后续查询与解释。  
FR19: 玩家可以查看已拥有幻兽的基础信息与当前状态。  
FR20: 玩家可以将一期已获得的幻兽纳入默认或当前出战队伍。  
FR21: 玩家可以对幻兽执行一期开放的基础养成动作。  
FR22: 玩家可以感知幻兽成长对战斗表现或战力变化的影响。  
FR23: 系统可以限制一期未开放的深层养成分支，保持幻兽系统范围可控。  
FR24: 玩家可以围绕幻兽形成持续投入的成长目标，而不是一次性使用后失去意义。  
FR25: 玩家可以进入一期基础副本或基础关卡内容并发起挑战。  
FR26: 玩家可以完成一期最小可玩战斗循环，并获得明确的胜负结果反馈。  
FR27: 系统可以根据战斗结果向玩家发放对应奖励或推进结果。  
FR28: 玩家在战斗失败后可以获得清晰的后续建议，例如返回养成、整理背包或再次挑战。  
FR29: 玩家可以看到自己的推进进度，并理解下一阶段可挑战内容。  
FR30: 系统可以把战斗、奖励、成长三者连接成可重复推进的主循环。  
FR31: 玩家可以通过战斗奖励、基础资源和养成动作持续提升角色或队伍强度。  
FR32: 玩家可以识别一期中的主要资源来源与主要资源去向。  
FR33: 系统可以在关键成长节点向玩家反馈成长结果，帮助玩家理解变强原因。  
FR34: 玩家每次上线时可以看到可继续执行的核心成长目标，而不是停留在无目标状态。  
FR35: 系统可以围绕一期内容形成连续的“获取资源 -> 执行成长 -> 再次挑战”的节奏。  
FR36: 运营人员可以调整一期初始化内容、基础奖励和核心玩法开放节奏。  
FR37: 运营人员可以控制一期核心入口的展示状态与引导节奏。  
FR38: 客服或排障人员可以查询玩家在初始化、奖励、容量阻断与基础结算上的关键记录。  
FR39: 客服或排障人员可以基于系统记录解释玩家为何获得、未获得或暂时无法领取某项收益。  
FR40: 系统可以为一期关键行为保留可追踪的结果记录，支持稳定迭代和问题定位。  
FR41: 玩家可以在网页端与微信小程序壳中体验一致的一期核心玩法流程。  
FR42: 平台宿主可以为共享 H5 提供进入游戏所需的宿主能力接入，而不改变核心玩法内容。  
FR43: 系统可以保证一期核心规则、成长结果与奖励结果在不同宿主中保持一致。  
FR44: 系统可以把旧 `htmlgame` 中可复用的页面表现、素材映射和行为参考转化为新一期产品输入，而不要求沿用旧工程结构。  
FR45: 系统可以在不破坏一期正式骨架的前提下，为二期系统预留可扩展边界。

### NonFunctional Requirements

NFR1: 玩家从进入游戏到完成一期主界面可交互的首屏加载，在常见移动网络环境下应控制在 5 秒内，弱网环境下应给出明确加载状态与重试反馈。  
NFR2: 一期核心交互页面，包括主界面、背包、幻兽页和基础战斗入口，在已有账号数据前提下的主要操作响应时间应控制在 2 秒内。  
NFR3: 战斗结算、奖励结算和养成确认等关键动作，在正常服务状态下应在 3 秒内返回明确结果或失败提示。  
NFR4: 一期客户端在移动端竖屏环境中必须保持可用，不得因页面结构过重导致核心流程无法操作或频繁卡死。  
NFR5: 角色初始化、初始资源发放、初始幻兽生成与默认队伍创建必须具备幂等保障，同一账号重复触发不得产生重复收益。  
NFR6: 战斗结算、奖励发放、资源扣减和养成消耗必须保证结果一致，不得出现客户端显示成功但服务端未落账的长期不一致状态。  
NFR7: 当结算、奖励领取或容量校验失败时，系统必须返回明确失败结果，并保持玩家资产处于可解释、可恢复状态。  
NFR8: 每日重置、次数恢复和其他时间相关逻辑必须统一基于服务端时间，网页端与微信小程序壳不得出现时间分叉。  
NFR9: 一期核心链路应达到月度 99.5% 以上的可用性目标，特别是登录进入、角色读取、战斗结算和奖励发放链路。  
NFR10: 账号身份、角色资产、幻兽数据、战斗结果与资源变更必须由服务端权威管理，客户端不得作为最终可信来源。  
NFR11: 网络传输中的账号会话信息与关键业务数据必须采用安全传输机制，防止明文暴露与基础篡改风险。  
NFR12: 玩家敏感数据与关键资产数据在存储时必须具备基础保护措施，避免因日志、配置或数据库误用导致泄露。  
NFR13: 一期即使不接入支付，也必须为后续 VIP、商城、订单与高价值资产审计预留安全边界，不允许当前设计阻断后续商业化接入。  
NFR14: 运营、配置与排障相关能力必须区分权限边界，避免非授权人员修改初始化、奖励或关键成长配置。  
NFR15: 共享 H5 必须作为唯一核心游戏应用，网页端与微信小程序壳仅承担宿主接入职责，不得复制或分叉核心玩法逻辑。  
NFR16: 一期核心流程，包括初始化、主界面导航、背包读取、幻兽养成、战斗与奖励反馈，在网页端与微信小程序壳中必须保持一致结果。  
NFR17: 平台差异能力必须限制在登录态接入、宿主能力桥接、分享占位、支付占位和存储桥接等边界内，不得渗透到玩法规则层。  
NFR18: 一期客户端必须兼容主流移动端竖屏 WebView 与常见桌面浏览器访问场景，保证核心流程可完成。  
NFR19: 一期必须具备最小配置能力，配置变更应能在不改动客户端主逻辑的前提下调整初始化内容、核心奖励与入口开放节奏。  
NFR20: 初始化、奖励、容量阻断、战斗结算和关键资源变更必须保留可查询记录，支持排障、申诉解释与后续运营分析。  
NFR21: 当核心链路发生异常时，系统必须能提供可定位到账号、角色、动作类型和结果状态的最小追踪信息。  
NFR22: 一期的日志、配置与追踪能力必须足以支撑小规模上线验证，而不是等到二期再补基础排障能力。  
NFR23: 一期数据模型与能力边界必须支持后续接入聊天、交易、联盟、VIP、商城及更深成长系统，而不要求推翻现有主循环结构。  
NFR24: 旧 `htmlgame` 中保留参考的页面表现、素材和映射数据，必须通过可替换的资源与配置方式接入，不得把旧原型代码结构固化为正式基础。  
NFR25: 在一期验证期内，系统应支持至少千级注册用户规模和百级同时在线量级的稳定运行；超过该范围时，应优先保证结算与资产正确性，再处理非关键体验退化。  
NFR26: 任何二期能力接入都不应破坏一期已验证的初始化、战斗、奖励和成长核心链路。

### Additional Requirements

- Epic 1 Story 1 必须先初始化 `Nx workspace`，采用 `React/Vite` 共享 H5 基线。
- 后端正式形态为 `NestJS 11` 模块化单体，不继续沿用旧 Express demo。
- 主数据库使用 `MySQL 8.4 LTS`，迁移与数据访问基线使用 `Prisma 7.6.0`。
- 一期 API 契约统一为 `REST + JSON + /api/v1`，并输出 OpenAPI/Swagger。
- 共享契约必须集中在 `shared contracts / schemas / types`，前后端不得各自定义一套接口类型。
- 认证采用“宿主身份输入 + 服务端统一会话令牌”模型。
- 前端状态边界固定为：`TanStack Query` 管权威服务端状态，`Zustand` 只管本地 UI/桥接状态。
- 平台差异必须封装在 `platform-bridge`、`web-adapter`、`wechat-adapter` 内，不得渗入业务页面。
- 一期核心模块必须按领域拆分，不能按旧原型页面脚本方式堆叠。
- 旧 `htmlgame`、旧素材 ID、旧 schema 只作映射输入，不能直接成为新系统业务实现。
- 一期必须纳入最小配置、审计日志、初始化/奖励/容量阻断/结算追踪能力。
- 一期部署形态固定为“静态 H5 + 容器化 API + 托管 MySQL”。
- Redis、消息队列、微服务、GraphQL、完整运营后台都属于后续阶段，不应写进一期阻断 story。
- 第一批实现优先级固定为：工作区初始化 -> 共享 contracts/platform bridge -> 初始化主链路 -> 核心玩法域。

## Epic 1: 玩家进入游戏并完成角色开局

玩家可以进入《召唤之王》、完成账号与角色初始化、获得初始资源与初始幻兽，并在主界面建立“我已经可以开始养成和推进”的认知。

### Story 1.1: 基于 Nx starter 初始化正式工作区与共享骨架

As a 开发团队,
I want 按已批准架构初始化 `Nx` 工作区、`game-h5`、`game-server`、共享 contracts 与 platform bridge 骨架,
So that 后续玩家开局能力可以在统一边界上实现而不再依赖旧原型结构。

**Acceptance Criteria:**

**Given** 已批准 starter 为 `Nx workspace + React/Vite` 共享 H5 基线
**When** 团队执行正式工程初始化
**Then** 生成可安装依赖并可执行基础 workspace 任务的正式工作区
**And** 工作区中创建 `game-h5`、`game-server` 与共享库边界

**Given** 新工程骨架已建立
**When** 检查共享能力目录与入口边界
**Then** 至少存在 `shared contracts`、`schemas/types`、`platform bridge` 的正式位置
**And** 不继续以 `htmlgame/game.html` 或旧全局脚本作为正式入口

**Given** 架构要求的一期边界
**When** 初始化目录结构与首批模块
**Then** 一期核心域目录可见
**And** 聊天、交易、联盟、VIP、商城等延期系统未被实现为一期业务模块

### Story 1.2: 建立统一会话与宿主接入边界

As a 玩家,
I want 无论从网页端还是微信小程序壳进入都能建立统一会话,
So that 我可以以一致方式进入游戏而不受宿主差异影响。

**Acceptance Criteria:**

**Given** 玩家从网页端进入共享 H5
**When** 客户端请求建立游戏会话
**Then** 系统通过网页宿主适配层接入身份信息
**And** 后端返回统一玩家会话结果

**Given** 玩家从微信小程序壳进入共享 H5
**When** 客户端请求建立游戏会话
**Then** 系统通过微信宿主适配层接入身份信息
**And** 后端返回与网页端一致语义的统一玩家会话结果

**Given** 宿主能力存在差异
**When** 共享 H5 调用宿主桥接
**Then** 业务页面只通过 `platform-bridge` 访问宿主能力
**And** 不在页面逻辑中直接写平台专属分支规则

### Story 1.3: 实现角色初始化与初始发放主链路

As a 新玩家,
I want 第一次进入时自动完成角色初始化并获得初始资源与初始幻兽,
So that 我可以立即开始一期核心玩法。

**Acceptance Criteria:**

**Given** 账号首次进入且尚未完成初始化
**When** 后端执行初始化流程
**Then** 系统一次性创建角色、初始资源、初始幻兽与默认队伍
**And** 初始化结果由服务端权威落账

**Given** 同一账号重复进入或网络重试
**When** 再次触发初始化流程
**Then** 系统不会重复发放初始收益
**And** 返回同一个已存在角色的当前状态

**Given** 初始化成功

**When** 业务页面触发登录、存储或分享相关能力
**Then** 页面只通过 `platform-bridge` 调用适配器
**And** 不在业务功能中散落平台特化逻辑

### Story 5.4: 建立旧原型素材与 ID 的正式映射接入层

As a 开发团队,
I want 通过统一映射层接入旧 `htmlgame` 的素材、页面表现参考和 ID 信息,
So that 一期业务模块可以复用现有资产而不继承旧工程结构。

**Acceptance Criteria:**

**Given** 主界面、背包或幻兽等一期模块需要使用旧素材或旧 ID 参考
**When** 模块读取相关资源
**Then** 相关信息通过正式映射表、资源注册表或参考数据层进入新骨架
**And** 业务模块不直接依赖旧页面脚本结构或旧 DOM 逻辑

**Given** 新系统消费旧素材映射
**When** 开发团队在业务模块中接入这些资产
**Then** 业务代码只依赖映射层输出的稳定键名、ID 或资源路径
**And** 不直接引用 `htmlgame` 的原始业务实现路径

**Given** 后续需要替换素材或调整映射
**When** 团队更新映射关系
**Then** 修改集中发生在映射层
**And** 至少一个已接入的一期模块无需修改核心领域逻辑即可继续工作

### Story 5.5: 固化延期系统入口规则与未来扩展占位边界

As a 产品与开发团队,
I want 为延期系统定义统一的入口状态、路由策略和模块占位规则,
So that 一期可以稳定交付，同时不阻断后续二期扩展。

**Acceptance Criteria:**

**Given** 聊天、交易、联盟、VIP、商城等系统不属于一期范围
**When** 团队组织当前产品模块、页面入口与路由
**Then** 这些系统被隐藏、置灰或标记未开放
**And** 不暴露可执行的半成品页面、接口或跳转路径

**Given** 架构需要支持后续扩展
**When** 当前系统定义模块边界、契约和占位规则
**Then** 延期能力以独立模块边界或显式占位方式保留
**And** 不要求一期核心域提前依赖这些未开放模块

**Given** 玩家接触到延期系统相关位置
**When** 页面展示入口或未开放说明
**Then** 系统使用统一的“未开放”表达和阻断反馈
**And** 不制造“功能已经可用”或“点击后才发现不可用”的误导体验
