# 面向 AI 代理的项目上下文

_本文件汇总 AI 代理在本项目中实施规划、架构和代码时必须遵守的关键规则与模式，重点记录不明显但最容易做错的事项。_

---

## 技术栈与版本

### 当前参考栈

- 旧参考客户端位于 `htmlgame/`，是面向移动端竖屏的 H5 单页原型。
- 旧客户端技术栈为：静态 HTML、jQuery、MUI CSS/JS、全局脚本结构、图片资源驱动 UI。
- 旧服务端技术栈为：Node.js `>=18`、Express `4.19.2`、mysql2 `3.9.7`、dotenv `16.4.5`、cors `2.8.5`。
- 旧数据库目标为：MySQL `8.x`，schema 位于 `htmlgame/db/schema.sql`。
- 当前主要源材料包括：
- 主需求基线：`召唤之王详细需求文档.pdf`
- 系统规则文档：`召唤之王/`
- 移动端竖屏布局参考：`布局图/`
- 旧 UI 与素材映射来源：`htmlgame/client/`

### 新构建基线

- 旧 `htmlgame` 技术栈仅作参考材料，不是长期正式生产架构基线。
- 新项目目标是共享 H5 主客户端，并明确区分网页容器与微信小程序壳的集成边界。
- PRD、架构和实现文档必须区分：
- 参考原型行为与素材
- 新生产骨架与正式模块边界
- 除非后续产物明确标记为保留，否则旧前端代码、旧 Express API 形态、旧 MySQL 演示 schema 都只视为调研输入。

## 关键实施规则

### 语言与代码模式规则

- 不得把 `htmlgame/client/game.js` 视为可持续演进的正式生产代码；它只是旧行为参考，不是新模块设计样式。
- 除非任务明确标记为“原型维护”，否则不得继续把正式功能直接堆到 `htmlgame/game.html` 或旧的全局脚本流程里。
- 旧浏览器端代码依赖全局变量、DOM id、内联 `vt` 指令路由和脚本加载顺序；新生产骨架只能继承行为语义，不能继承这种实现方式。
- 旧服务端代码使用 JavaScript、CommonJS 与 `async/await`；在新架构未定前如果需要写临时 Node 工具，优先保持 `async/await` 与显式错误响应，不要退回回调风格。
- 不得默认假设项目已经存在 TypeScript、打包器或前端框架约束；任何新工具链采纳都必须来自正式架构文档，而不是故事实现时临时决定。
- 定义领域模型、枚举、映射表和 UI 文案时，必须保留需求文档里的中文业务术语，不得擅自改写成泛化英文游戏词汇。
- 文档、接口、存储和素材元数据必须保持 UTF-8 与数据库 `utf8mb4` 兼容，避免中文内容损坏。
- 旧物品 id、素材 id、图标文件名等数值标识如果复用，必须通过映射层或配置层进入新系统，不能散落在新业务逻辑中。
- 任何 demo 级行为不得直接继承进正式规则，包括仅用户名登录、内嵌本地凭据、浏览器端假数据捷径等。

### 结构与平台边界规则

- 新生产构建必须清晰分离 `共享游戏应用`、`平台壳适配层`、`后端服务契约`，不得把职责混在同一层。
- 共享 H5 主客户端是主要产品载体，负责游戏页面、导航流转、领域展示、战斗表现和本地交互状态。
- 网页容器与微信小程序壳只是交付外壳；它们可以提供登录桥、存储桥、支付桥、分享桥和生命周期钩子，但不能承载游戏规则或特性专属 UI 逻辑。
- 所有平台差异都必须封装在适配器接口之后，确保同一套 H5 游戏流程能以最小分歧运行在浏览器和小程序壳中。
- 不得用旧的 `vt` 指令路由、DOM tab 切换或直接全局事件绑定作为新客户端的结构基础。
- 不得把旧 Express 路由布局或 demo schema 当成正式服务契约；新 API 边界必须从一期产品范围和架构决策中推导。
- 旧 `htmlgame` 页面只可用于页面清单、布局参考、术语映射和素材检索，不能据此认定新系统必须保留相同页面组成、导航层级或功能归组。
- 新模块划分必须围绕一期可玩版核心域展开：账号与角色初始化、主界面导航、背包、幻兽、基础战斗循环、基础资源成长。
- 聊天、交易、联盟、VIP、商城、运营后台等非一期模块必须显式延期或隔离，不能在一期模块设计中偷偷渗入。
- 当复用 `htmlgame/client/` 素材时，素材目录与映射元数据必须和游戏逻辑分离，避免未来换 UI 时牵连领域层重写。

### 测试规则

- 当前项目应视为尚未建立正式生产级自动化测试基线，不能因为旧 demo 能手动跑起来就声称已经稳定或有覆盖。
- `node_modules`、BMAD 工具目录或第三方包自带测试都不算《召唤之王》项目验证证据。
- 新一期可玩版测试优先验证领域正确性与契约稳定性，而不是截图数量或零碎 UI 细节。
- 第一批自动化测试优先覆盖：
- 账号与角色初始化流程
- 初始资源发放与默认状态创建
- 背包与幻兽获取、读取和状态变更
- 基础战斗循环输入、结算与奖励发放
- 基础资源成长计算与持久化
- 战斗公式、奖励分配、养成消耗、初始化默认值等领域规则，必须能在不依赖平台壳的情况下独立测试。
- 网页与微信小程序壳适配层应作为边界适配器测试，使用 mock 平台能力，不应在壳层验证核心游戏规则。
- 一期核心模块在扩展延期系统之前，必须先有 API 契约测试验证请求与响应结构。
- 当数值规则来源于需求文档、表格或映射资料时，优先采用 fixture 驱动或表驱动测试，便于显式比对规则变更。
- 对旧原型的维护任务，手工验证可以作为临时检查，但不能作为新生产骨架的长期验证策略。
- 后续任何 PRD 或架构文档，都应为每个一期模块先定义最低验证路径，再进入实现。

### 代码质量与风格规则

- 以清晰的领域边界优先于图省事的文件堆积，绝不能再次形成 `htmlgame/game.html` 加 `game.js` 这种巨型单入口。
- 游戏规则、配置数据、UI 组合、平台集成必须分层，不能把公式、素材路径、请求调用和渲染逻辑揉在同一模块中。
- 需求文档与规则说明是业务真相输入；当实现与旧原型行为冲突时，除非 PRD 或架构文档明确记录覆盖决定，否则以正式文档为准。
- 产品文档、映射表和用户可见文案必须保留中文功能名与玩家术语；如果后续引入英文内部标识，也必须能回溯到原中文概念。
- 任何复用的旧素材 id、物品 id、幻兽 id、地图 id、图标文件名都应通过专门的参考表或配置注册表进入系统，不能用零散硬编码。
- 新代码结构必须清楚体现“一期”与“延期”边界，避免把聊天、交易、联盟、VIP、商城等延期系统和一期核心玩法写进同一功能模块。
- 一旦架构选定具体技术栈，就应优先使用显式数据契约以及类型或 schema 校验边界，不得在模块之间传递无主的松散 payload。
- 注释只用于解释业务规则例外、映射决策和不明显的文档约束，不要给自解释代码增加噪声注释。
- 所有临时兼容桥接，无论面向旧素材还是旧数据，都必须被标注为过渡层并隔离放置，以便未来移除时不重写核心领域层。
- 正式代码路径中不得保留隐藏环境耦合，例如内嵌本地密码、固定 localhost 假设、硬编码 demo 端口。
- 新构建的目录和文件结构应反映产品域与平台边界，而不是照抄旧原型的页面堆叠方式。

### 开发流程规则

- 本项目的规划与实施必须遵循 BMAD 产物链：`project-context` -> `PRD` -> `architecture` -> `epics/stories` -> `implementation`，不得在只看旧原型后直接进入完整功能堆叠。
- 每个较大任务都应先声明其归属：
- 旧原型参考或维护
- 新生产骨架规划
- 新生产骨架实现
- 素材或规则映射
- 在设计或实现任何玩法系统前，必须显式列出其输入来源：
- `召唤之王详细需求文档.pdf`
- `召唤之王/` 下对应系统规则文档
- `布局图/` 下对应移动端布局参考
- 如适用，再补充 `htmlgame/` 下对应的旧行为或素材
- 当需求文档、规则文档、布局参考与旧原型发生冲突时，不得静默即兴决定，必须在当前规划产物中记录哪一份来源胜出。
- 一期范围是默认构建边界；聊天、交易、联盟、VIP、商城及其他后一期系统必须明确标记延期，不能习惯性地做半成品接入。
- 规划产物写入 `_bmad-output/planning-artifacts/`，实施周期产物写入 `_bmad-output/implementation-artifacts/`。
- `docs/` 只有在内容被明确整理并写入后，才视为长期项目知识库；不能默认其中已经有权威资料。
- 当前工作区不是 git 仓库，因此项目指导中不得依赖 branch、PR、commit 作为默认流程依据；应优先依赖明确的产物更新与范围清晰的文件改动。
- 本地 `.env` 中的值不得传播进文档、示例代码或架构决策，应改用配置契约与占位示例。
- 任何一期模块在开始实现前，PRD 与架构文档都必须明确说明：
- 它是否属于一期范围
- 它复用了哪些旧素材或数据参考
- 哪些行为是权威实现，哪些只是 demo 参考
- 如果某个延期功能仍需要占位命名、导航入口或未来兼容设计，必须显式记录边界，不能先埋隐藏的半逻辑。

### 关键红线规则

- 一期的账号与角色初始化必须由服务端权威完成，且具备幂等性；重复进入、重试或壳重启都不能生成重复角色、重复新手奖励或不一致默认状态。
- 所有战斗结算、资源消耗、奖励发放和成长推进都必须由可信后端规则裁定；共享 H5 与小程序壳只能展示和提交动作，不能成为最终裁定者。
- 所有时间相关系统必须使用统一服务端时间模型，浏览器时间或小程序客户端时间都不能决定重置、冷却、成熟、完成或奖励可领状态。
- 资源系统必须保留明确的来源与去向语义；即使是一期，也不能用模糊、不可追溯的临时修改直接更新核心货币与成长资源。
- 高价值资源变动、新手发放和核心成长奖励必须预留日志或审计能力的设计路径，即便一期实现保持精简。
- 背包、幻兽栏及其他一期保留的存储容量必须独立建模，不能为了省事合并成一个泛化容器而破坏后续规则。
- 满仓或满栏状态必须明确阻断并提示，不能静默丢弃奖励、覆盖库存或自动销毁实体，只因为旧原型处理得比较松。
- 旧原型和完整首发版文档中的全量导航，不能反向绑架一期范围；如果某入口延期，就应被明确标为延期，而不是埋入隐藏半功能。
- 平台壳集成不得导致玩法规则分叉；网页端与微信小程序可以在宿主 API 上不同，但战斗公式、成长逻辑和功能边界必须保持一致，除非有明确文档说明。
- 绝不能把本地 demo 便利手段当成正式事实，包括硬编码 localhost 地址、内嵌凭据、离线假数据、宽松 CORS 假设或仅用户名鉴权。

## 使用说明

**对 AI 代理：**

- 开始任何规划或实现前先阅读本文件。
- 如与更新后的 PRD、架构文档或已记录决策冲突，以更新后的正式产物为准。
- 遇到一期与延期边界不清时，优先采取更保守的范围解释。
- 当项目出现新的稳定规则、平台边界或映射约束时，及时更新本文件。

**对人工维护者：**

- 保持本文件精简，只记录 AI 代理容易遗漏或误判的关键约束。
- 技术栈、平台边界、一二期范围或核心规则变化时同步更新。
- 当某条规则已经在架构、代码结构或工具约束中显性固化后，可从本文件删除冗余表述。
